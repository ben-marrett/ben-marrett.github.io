Classes vs. ID's - best practices.

There are times and places where either class or ID could be used as a selector to style our content, and knowing which is the more appropriate option in any given scenario can make all the difference for reasons of organization, readability and developer collaboration.

A wee analogy.

One way I have come to think of classes and ID's is a simple comparison with my own young family at home. Our children all each have a unique ID, a collection of individual attributes that make up each child. Fingerprint, appearance, personal characteristics etc. On the other hand, our children also go to a school, where there are certain rules that apply to everyone who attends, irrespective of what their name is or where they come from or some other unique personal factor. This CLASS may create rules such as: everyone within this room is between the age of 9 & 10, or, everyone in this class will wear this navy blue uniform, or, everyone in this class will do mathematics at 10am on a Tuesday.
So, for me this presents a simple way of looking at how to use Class and ID. If there is a particular part of a website that I may want to narrow in on and make specific changes to in a way unlike any other, then I would use the ID. This may be the equivalent of picking out a student in the school whom I would like to offer some unique opportunity to, perhaps because they show a proficiency in music for example. I can pick the ID of this student and add in this special curriculum element for them that doesn't affect any of the other students in the school. Same thing on a home level - we can select a child and decide "this kid needs a haircut!", and make this change to one particular child, where it would really not be appropriate to go and insist that the entire school class should have a short back and sides.
On the class level we may decide "Hey, let's get everyone a new uniform", and then it is appropriate to apply the rule to everyone, that from then on every student in the class will arrive with a red uniform. Or, everyone in the class will now do gardening at 10am on Tuesdays instead of mathematics.
(I don't know that treating kids like elements to be controlled and shaped is ideal but initially it made sense!)

Best practices.

Some best practices within the use of these two selectors, include:
  • Name semantically. Don't be too vague.
    Describe the context of the element rather than just the position on screen or what it looks like. The name could answer - what is it? what does it do? why is it here?

  • ...but don't be too specific either.
    For example, if we name a Class with a specific color and later change the color, then we will be going back to the html to change this color title in the class otherwise it will be named confusingly from then on.

  • Align with other websites.
    What are other developers naming similar elements?

  • Is being able to link directly to a certain section on a page important?
    If so, ID's are unique in allowing us to do this.

  • Is there more than one element that requires this same styling, now or in future?
    If this is even a possibility, Class is probably the desired option.
