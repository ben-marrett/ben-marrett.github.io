Problem Solving

Houston...

Bumped into a few things all at once during this sprint. It all really piled on at the section on 'Built-In Methods'. This was a challenge where we were tasked with making use of functions which are already written within the JavaScript library and available to use. As our intro did state, "you just need to know how to use it!"

The problems probably started with the realisation that for this challenge, the responsibility of research was upto me. Some initial documents were given as a starting point, but really I was gonna have to put in the work to find what was needed and bring it all together.
So, here I am with only the bare bones of a function in front of me, needing to work out how to write code that will pass certain tests. My initial attitude toward looking for ways to go about solving these tasks was pretty serious, and laden with panic. So from the get go I was really building stress and confusion.

Scouring the internet led to finding many different potential ways of writing the same thing. Some ways that were overly advanced for the task at hand, some that were too simple and insufficient. I think it was the lack of clarity on exactly what I was aiming for that contributed to solidifying the problem.

In essence, the problem was to have the function look within an object and locate a name and the door number associated with that name. However I had taken the story and dramatized it... Suddenly the code is a complicated game of hide and seek and I've got to find Scooby Doo... and how I am supposed to do that!? and all of a sudden I am caught up in the mystery and in desperate need of help!

Widening back I could eventually see that it was time to get a bit more precise on my googling. So, how do I use a function to search within an array and locate a certain parameter? Rechecking the instructions I could see it was to use the built-in method "find". OK. That's a good start.

I had got to a point where I was reasonably confident that I was using "find" to succesfully locate Scooby, but just wasn't quite getting the return in the console that I wanted.
Why was it still coming back with an error about undefined?

This is when another look at the error message gave a vital key.

Instead of just seeing a big red "NO, WRONG!" sign, something clicked. One of the initial tests involved giving the function an empty array, and I realised this is where the 'undefined' was coming from. I had deleted what had allowed me to pass the initial 2 tests of the exercise to make way for my new code, instead of building on that code. So the solution was to bring back into the code what should happen if the array is empty. This one piece allowed everything to flow nicely.
Valuable lessons learnt and further understanding gained on problem solving.

A bit more elegant.

A time that I solved something a bit more elegantly is when working on a task called FizzBuzz. For this one I had everything working nicely until one point where I converted all inputs into lower case so that I could find the word "buzz" regardless of how it was put in eg. "BuZZ/buzZ/BUZZ". The trouble here was that by doing this I had affected a previous test where some words like "Hello" needed to start with a capital and were now lower case along with all the buzzes.

I sorted this quite quickly by adjusting the code to only temporarily put an element into lower case and check if it was equal to "buzz", and from there it would remove this word and everything else would be unaffected by the lower case change. I was pretty chuffed with this one.

Problem Solving Techniques and Processes

  • I realise that I did not fully have a grasp of the value of pseudocode through this sprint. I was using it vaguely and not in a useful manner. Now I begin to see how useful and appropriate it can be in saving time and frustration, particularly coding for tests.
  • Trying things out is fine but for me can lead to increasesd frustration if results do not come.
  • I love the rubber ducky method and feel confident in using it. It's surprisingly insightful!
  • Error messages are invaluable and oft times provide subtle hints at simple mistakes and easy solutions.
  • Console.logging is something I will aim to use much more productively.
  • Googling is fine... can continue to refine search and narrow in on exactly what is needed. Tune out the noise.
  • Asking peers for help - I have not been overly confident in this and tend to look to Slack in the hope that others have asked for help. I am sure the following sprint may offer an opportunity to remedy this.
  • I am ok asking coach for help when really stuck.
  • Reflection is great and helps see all from a wider perspective of what's really going on.
